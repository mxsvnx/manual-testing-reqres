# Баг-репорты — форма авторизации Reqres.in

Тестирование проводилось вручную через Postman v10.
Дата: 2024-06
API: https://reqres.in

---

## BUG-001 — Одинаковое сообщение об ошибке для разных проблем с email

| Поле | Значение |
|------|----------|
| **ID** | BUG-001 |
| **Статус** | Open |
| **Приоритет** | Medium |
| **Severity** | Minor |
| **Компонент** | API / Авторизация |
| **Эндпоинт** | POST /api/login |

**Описание:**
API возвращает одно и то же сообщение `"user not found"` для двух разных ситуаций: несуществующий пользователь и невалидный формат email. Клиентская сторона не может различить эти случаи и показать пользователю корректное сообщение.

**Шаги воспроизведения:**

Случай А — несуществующий пользователь:
```
POST /api/login
{"email": "notexist@example.com", "password": "anypassword"}
```

Случай Б — невалидный формат email:
```
POST /api/login
{"email": "eveholtreqresin", "password": "cityslicka"}
```

**Фактический результат (оба случая):**
```json
HTTP 400
{
  "error": "user not found"
}
```

**Ожидаемый результат:**
- Случай А: `{"error": "user not found"}`
- Случай Б: `{"error": "Invalid email format"}`

**Примечание:** Поведение может быть намеренным из соображений безопасности (не раскрывать, зарегистрирован ли email). Требует уточнения у разработчика.

## BUG-002 — Отсутствует валидация формата email

| Поле | Значение |
|------|----------|
| **ID** | BUG-002 |
| **Статус** | Open |
| **Приоритет** | Medium |
| **Severity** | Major |
| **Компонент** | API / Авторизация |
| **Эндпоинт** | POST /api/login |

**Описание:**
API принимает строку без символа `@` в поле email и обрабатывает её как корректный запрос — возвращает 400, но с ошибкой `"user not found"`, а не `"invalid email"`. Валидация формата email на стороне сервера отсутствует.

**Шаги воспроизведения:**
```
POST /api/login
{"email": "eveholtreqresin", "password": "cityslicka"}
```

**Фактический результат:**
```json
HTTP 400
{
  "error": "user not found"
}
```

**Ожидаемый результат:**
```json
HTTP 422
{
  "error": "Invalid email format"
}
```

## BUG-003 — Непоследовательная обработка пустых значений и пробелов в поле password

| Поле | Значение |
|------|----------|
| **ID** | BUG-003 |
| **Статус** | Open |
| **Приоритет** | Low |
| **Severity** | Minor |
| **Компонент** | API / Авторизация |
| **Эндпоинт** | POST /api/login |

**Описание:**
При передаче пустой строки в поле `password` API возвращает `"Missing password"`. При передаче строки из пробелов поведение должно быть идентичным, однако явной обработки этого случая нет, что может привести к непоследовательным ответам.

**Шаги воспроизведения:**

Случай А:
```
POST /api/login
{"email": "eve.holt@reqres.in", "password": ""}
```

Случай Б:
```
POST /api/login
{"email": "eve.holt@reqres.in", "password": "   "}
```

**Фактический результат:**
- Случай А: `HTTP 400 {"error": "Missing password"}`
- Случай Б: `HTTP 400 {"error": "Missing password"}`

**Ожидаемый результат:**
Оба случая должны явно указывать, что пустое или состоящее из пробелов значение недопустимо. Поведение зафиксировано для документирования — требует проверки на продакшн-API.
